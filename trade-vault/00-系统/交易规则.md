# 交易规则

## 当前边界

- 当前没有实盘券商接口，所有真实交易必须人工确认。
- 系统可以生成交易建议、模拟盘订单和风控提醒，但不能绕过人工确认。
- MySQL 是交易事实源；vault 只写人读摘要和复盘结论。
- Agent 只通过 `bin/trade` 或 MCP 操作系统。

## 每日流程

| 时间 | 动作 | 产物 |
|---|---|---|
| 盘前 | 检查系统健康、市场状态、候选池、今日决策 | `04-决策/今日决策.md` |
| 盘中 | 轮巡持仓、价格、风控事件 | MySQL 事件流 |
| 收盘后 | 汇总当天所有操作和系统日志 | `02-巡检/YYYY-MM-DD.md` |
| 周末 | 汇总交易表现和系统问题 | `03-分析/周复盘/` |
| 月末 | 汇总策略、风控、执行质量 | `03-分析/月复盘/` |

## 人工确认

任何真实买卖都必须进入人工确认流程：

```bash
bin/trade manual-trades list --json
bin/trade manual-trades confirm REQUEST_ID --json
bin/trade manual-trades reject REQUEST_ID --json
```

确认前至少检查：

- 交易理由是否来自当前有效策略和候选池。
- 风控是否允许新增或减少该仓位。
- 当前现金、持仓数、单票仓位、组合暴露是否满足规则。
- 当天是否已有异常数据源或 pipeline 失败。

## 选股入口

日常选股通过 CLI 或 MCP 触发，结果进入候选池和事件流：

```bash
bin/trade screener run --query "近20日放量突破，市值100亿以上" --json
bin/trade screener candidates --json
bin/trade screener promote CODE --to core --json
bin/trade screener reject CODE --reason "原因" --json
```

选股结果不是交易指令。进入核心池或观察池后，仍需经过评分、风控、今日决策和人工确认。

## 巡检报告要求

每日巡检报告必须短而完整，避免复制原始事件。推荐结构：

```markdown
# 收盘巡检 - YYYY-MM-DD

## 结论

## 今日操作

## 系统运行

## 交易与风控

## 数据质量

## 待办
```
